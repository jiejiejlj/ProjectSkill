# interview-me 盘问方法论

本文件是 SKILL.md 的展开,讲清「怎么穷追式盘问」。源出 grilling 技能,按本仓库场景做了增强。

## 核心原则(不可妥协)
1. **穷追不舍**:对计划/设计的每个角落都要问透,默认不偷懒、不替用户跳过,直到达成共识;不主动提供快捷退出。
2. **一次一问**:一口气抛多题会让人懵。问完一题,停下等反馈,再问下一题。
3. **每题给推荐**:每个问题都附「我的推荐答案 + 简短理由」,别只抛问题让用户裸答。
4. **代码库优先**:凡能靠读代码 / 搜仓库查到的答案,先查,别拿去问用户。

## 决策树怎么走
- 把计划/设计拆成一棵决策树:每个待定点是一个节点,节点间有依赖。
- **按依赖排序**:先问上游决策——它的答案会改变下游有哪些选项、甚至下游是否还存在。
- **逐一深入**:沿一条分支问到底,再回到下一条分支;别在多条分支间来回跳,以免依赖错乱。
- 上游答案敲定后,据此裁剪 / 重排下游问题(有的下游可能因此消失或新增)。

## 提问机制
- **能枚举选项**:用 AskUserQuestion,一次一问;推荐答案放第一个选项并标注「(推荐)」。
- **开放题**(枚举不出选项):用纯文本提问,问题末尾写明「我的推荐:……(理由)」。
- 无论哪种,都只问一题就停,等反馈再继续。

## 收尾:共识小结
盘问到共识达成后,输出一份结构化小结:
- **已定决策**:逐条列「问题 → 敲定的答案」。
- **仍未决项**:还没谈拢或刻意延后的点。
- **下一步**:据共识接下来该做什么。

然后**主动问用户**:要不要把这份小结写成文件?写到哪个路径?——用户要才落盘,默认不落盘、不污染仓库。

## 反例(别这么干)
- ✗ 一条消息里抛 3 个问题。
- ✗ 只问「你想怎么设计?」却不给推荐。
- ✗ 能 grep 出来的事实拿去问用户。
- ✗ 上游还没定就先纠缠下游细节。
- ✗ 用户没喊停就自己草草收尾。
