# interview-me-with-docs 方法论

本技能 = **interview-me(盘问底座)+ 领域建模(活文档层)**。盘问纪律以 interview-me 技能为准,这里只讲叠加上去的领域建模部分。

## 叠加关系
- **底座**:interview-me 的全部纪律——一次一问、每题给推荐、按依赖逐层、能查码先查、穷追到共识、不设退出口。
- **增量**:盘问过程中持续维护领域模型,并把成果当场落成文档。

## 领域建模五条纪律(盘问中持续执行)
1. **拿术语表对质**:用户用词和已定义术语冲突时立刻拍下——「CONTEXT.md 里 X 指 A,你这像是 B,到底哪个?」
2. **逼精模糊措辞**:出现含糊/超载的词,提出一个精确的规范术语,并列出要舍弃的近义词。
3. **造边界场景**:编具体边缘案例,逼用户说清两个概念之间的界线到底在哪。
4. **和代码对账**:把描述拿去和真实代码核对(`Grep/Glob/Read`),发现矛盾就摊开。
5. **当场落术语**:术语一谈定,立刻按 [context-format.md](context-format.md) 写进 `CONTEXT.md`。

## 文档结构与位置
- 落在**被设计项目的根目录**。
- **单上下文**:根目录一个 `CONTEXT.md`;`docs/adr/` 放 ADR。
- **多上下文**:根目录 `CONTEXT-MAP.md` 指向各「限界上下文」(各自的 `CONTEXT.md` + `docs/adr/`)。
- 判定顺序:先看有没有 `CONTEXT-MAP.md` → 再看根 `CONTEXT.md` → 都没有则懒创建。

## 懒创建与落盘时机
- **有内容要写才建文件**:解决第一个术语才建 `CONTEXT.md`;需要第一条 ADR 才建 `docs/adr/`。
- **首次落盘前确认一次**(放哪、建不建);之后随谈随写,不再每条都问。
- **收尾**给出本次产出/更新的文档清单(路径)。

## ADR:何时写
三闸全过才写(详见 [adr-format.md](adr-format.md)):① 难反悔 ② 没上下文会让人意外 ③ 是真权衡。缺一不写。

## 反例(别这么干)
- ✗ 把实现细节/取舍写进 `CONTEXT.md`(那是 ADR 的事)。
- ✗ 攒到最后才补文档(本技能要求随谈随写)。
- ✗ 给每个易反悔的小决定都写 ADR。
- ✗ 术语含糊就放过,不给规范词。
